3. 🗄️ MODÈLE DE DONNÉES DÉTAILLÉ (21 tables)

Tables principales

  • users (UUID, phone, type, status)
  • profiles (user_id, coordonnées, rating)
  • categories + services
  • provider_services (pivot prestataire/service)
  • rfqs (appels d'offres anonymes, lat/lng)
  • bids (propositions prestataires)
  • orders (commande finale + statuts)

Tables transactionnelles & support

  • transactions (traçabilité paiements)
  • wallets / withdrawals
  • reviews (notation, commentaires)
  • messages (chat entre utilisateurs)
  • disputes, notifications, geo_zones, subscriptions
🔗 Relations clés : users → profiles → provider_services → services → categories.
RFQ (client) ⇄ bids (prestataire) → orders → transactions / reviews. Toutes les migrations sont séquencées pour respecter les dépendances.

6. 🎯 SPÉCIFICATION DU MODULE RFQ (APPEL D'OFFRES ANONYME)

📌 Flux critique :
1️⃣ Client publie RFQ → anonymisé (client masqué).
2️⃣ Queue job: recherche PostGIS des prestataires dans le rayon → notifications FCM.
3️⃣ Prestataire consulte /rfqs/nearby → propose un prix/délai (POST /bid).
4️⃣ Client reçoit les offres, compare profiles → sélectionne un bid.
5️⃣ Génération de la commande (order) → paiement sécurisé.
✅ Protection contre la double sélection via verrouillage (Laravel lock).

Exemple payload RFQ (client) : { "service_id": "uuid", "description": "Réparation fuite eau", "lat": 0.4162, "lng": 9.4673, "preferred_date": "2026-06-10" }
